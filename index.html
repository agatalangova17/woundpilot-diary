February 8 – March 17, 2026 Proposal Phase

Proposal Submission, Supervisor Sign-Off & Project Framing

Milestone Research

Work Done

Emailed an initial draft of the project proposal to supervisor Bohuš Získal on 8 February for early feedback. Bohuš returned written feedback the following day asking me to clarify the project's intended outcome (proof of concept vs. MVP vs. finished product), tighten the research question accordingly, add GUI design to the areas for further investigation, present the plan as a Gantt chart with documentation activities included, and frame the deliverables in less product-specific language.

I revised the proposal across the following two weeks. The main change was reframing WoundPilot from a clinically-positioned product to an MVP, with appropriately softened deliverables language, a Gantt chart with documentation slots added, and GUI design listed as an area for further investigation. Submitted the revised proposal via Blackboard on 23 February.

Official sign-off feedback arrived around mid-March. Following sign-off, Bohuš and I had a Zoom call where he suggested reframing the project once more, this time from MVP to proof of concept, on the grounds that this framing better matches what the project can credibly deliver and assess. We also agreed to revise the research question to reflect the newer framing, and to introduce a concrete success criterion of ≥80% etiology agreement with specialist judgment across ten clinical test cases before the Progress Review. The updated research question became: "Can combining iPhone LiDAR-based wound measurement with rule-based clinical reasoning in a mobile application provide clinically useful outputs for chronic wound assessment?"

Around the same time, confirmed Dr. Jozef Lang, a wound care specialist practising in Slovakia, as clinical advisor and validation partner. His role would be to validate the diagnostic logic, and give final usability feedback.

March 2 – 8, 2026 Week 1

Project Kickoff, Core Architecture & Patient Management

Development Milestone

Work Done

With the project proposal approved, development began in earnest. Building on the market research conducted during the proposal phase — which identified a clear gap in digital wound assessment tools for non-specialist community healthcare providers — I set up the full technical foundation for WoundPilot. The Xcode project was created with SwiftUI targeting iOS 17+, and the Firebase backend was configured with Authentication, Firestore, and Storage services.

The initial project structure was established with clean separation between models, views, and services. I implemented the welcome screen, login/registration flow with form validation, a forgot password view, and the initial Terms and Conditions and Privacy Policy screens — essential for a healthcare application from day one.

Moving quickly into core functionality, I designed and implemented the Firestore data models for patients, wound groups, and individual wound records using a hierarchical structure: users → patients → wound groups → wounds. Full CRUD operations were built for patient records. The wound history view was implemented with Firestore persistence, a healing trajectory graph was added using SwiftUI, and wound image capture was built with dual-source support — camera or photo library. A wound grouping system was created to organise wounds by anatomical region, and a quick analysis mode was added for rapid assessments without full patient registration.

Problems Encountered

Firebase security rules required careful design to ensure complete data isolation between clinician accounts. Designing the Firestore document structure meant balancing query efficiency with logical data organisation — nested subcollections add complexity to both queries and cascade deletions. I also evaluated whether to use machine learning or a rule-based system for the clinical reasoning engine.

Solutions Found

Implemented Firestore security rules using the request.auth.uid matching pattern for per-user data isolation. Adopted a hierarchical Firestore structure mirroring the clinical workflow, which my clinical advisor confirmed matches how wound records are organised in practice. Chose a deterministic rule-based engine over ML for transparency, GDPR compliance, and explainable outputs.

Personal Development

The market research during the proposal phase showed that while wound care apps exist, very few combine measurement, diagnosis, and treatment recommendations in a single tool — and none offer LiDAR-based measurement on curved surfaces. Early consultation with my clinical advisor shaped the data architecture significantly, particularly the wound grouping approach. Discussed the rule-based vs. ML decision with my supervisor, who agreed it was well-justified.

March 16 – 22, 2026 Week 3

Questionnaire System Overhaul & Rule-Based Analysis Engine

Development Milestone Research

Work Done

A fundamental shift in the application's clinical logic. The questionnaire was completely rebuilt from a simple static form into a conditional branching system powered by a rule-based analysis engine. Instead of presenting all 50+ possible questions, the system uses conditional logic to show only relevant questions based on the patient's profile and previous answers — typically presenting 15–25 targeted questions per assessment.

The questionnaire architecture was restructured into clean components: QuestionnaireSteps, QuestionnaireOptions, and QuestionnaireContext. The context object tracks all answers and drives which questions appear next, mirroring how clinicians narrow their differential diagnosis through targeted questioning.

Initial work began on the AR measurement view using ARKit and RealityKit — setting up the camera feed and basic surface detection as groundwork for LiDAR-based wound measurement. Security was improved by removing the Firebase configuration file from version control.

Problems Encountered

Designing conditional logic that mirrors real clinical decision trees was complex — wound assessment involves multiple branching paths depending on location, appearance, patient history, and risk factors. The restructured questionnaire had ordering bugs where steps appeared in the wrong sequence.

Solutions Found

Built QuestionnaireContext as a central state manager that evaluates conditions for each step before presenting it. Fixed step ordering through a proper validation chain in QuestionnaireSteps.swift. Researched the T.I.M.E. framework extensively to inform the question flow design.

Personal Development

Extensive research into the T.I.M.E. wound assessment framework (Tissue, Infection/Inflammation, Moisture, Edge) and clinical decision trees for wound care. Understanding these frameworks was essential for designing a questionnaire that produces clinically meaningful outputs. Confirmed the approach with my supervisor.

Started roughing out the Introduction and Literature Review sections of the project report in parallel with development — capturing the rationale for the rule-based approach and the T.I.M.E. framework context while it was fresh. Writing alongside coding is proving much easier than trying to reconstruct decisions later.

March 23 – 29, 2026 Week 4

Anatomical Body Localization & LiDAR Measurement Integration

Development Milestone Bug Fix

Work Done

Implemented the anatomical body localization system — a critical component for wound location identification. This includes WoundLocationPickerView with interactive body diagrams, plus specialised HandDetailSheet and FootDetailSheet for detailed extremity selection. Wound location is clinically significant as it directly influences the rules engine's diagnostic pathways.

The LiDAR-based AR measurement system was integrated into the main workflow through ARMeasureView. The complete flow now follows: body localization → AR measurement → questionnaire → report. MeasurementFlowWrapper was created to manage navigation state across this multi-step process.

The dressing recommendation view was added as the assessment flow's final step. The healing graph was improved, and several questionnaire bugs were fixed to ensure consistent step progression. Deprecated .onChange(of:perform:) calls were replaced with iOS 17 syntax throughout.

Continued building out the project report in parallel — documented the architectural decisions around the assessment flow and began drafting the design section covering the data model and navigation structure. Recording these decisions now while they are fresh avoids having to reconstruct rationale later.

Problems Encountered

The multi-step flow created significant state management complexity. The body localization view wasn't dismissing properly when transitioning to AR measurement, causing navigation stack corruption. Questionnaire dismissal also had issues with state not resetting between assessments.

Solutions Found

Created WoundLocationPickerViewWrapper as a coordinator managing transitions between flow steps with proper dismissal and data passing. Fixed questionnaire state by resetting the navigation context between assessments. Replaced all deprecated .onChange calls.

Personal Development

Learned about the clinical significance of wound location in etiology determination — wounds on the medial ankle are often venous, while lateral ankle or toe wounds are more likely arterial. This understanding directly shaped the body localization UI and its connection to the rules engine.

April 13 – 19, 2026 Week 7

4-Point AR Measurement System & PUSH Score Implementation

Development Milestone Research

Work Done

The most technically challenging week of the project. I implemented a 4-point AR measurement system achieving ±0.3–0.5cm accuracy on flat surfaces and ±1cm on curved surfaces like knees and heels. The system uses multi-sample raycasting with 25 points around each tap location, outlier filtering, and weighted averaging. Extending LiDAR measurement to curved anatomical surfaces represents a novel technical contribution — most existing solutions only handle flat surfaces.

The PUSH (Pressure Ulcer Scale for Healing) scoring system was built in PUSH.swift and PUSHTrajectory.swift. This evidence-based tool scores wound area, exudate amount, and tissue type to track healing over time. The trajectory graph now shows PUSH scores across assessments, giving clinicians a visual healing indicator.

A comprehensive UI overhaul replaced emoji icons with SF Symbols for professional appearance. The smoking assessment was separated into its own section, and profile, contact, and patient editing views were refined.

Problems Encountered

Initial curved surface accuracy showed ±3–4cm errors. The PUSH score wasn't updating correctly with remeasurements, and manual measurements weren't syncing to the trajectory graph.

Solutions Found

Developed multi-sample raycasting: 25 rays per tap in a pattern, statistical outlier filtering, and weighted averaging — reducing curved surface error to ±1cm. Fixed PUSH trajectory by properly incorporating remeasurement data points and recalculating on each new assessment.

Personal Development

The AR work required deep understanding of both ARKit raycasting and geometric challenges of non-planar measurement. PUSH scoring research involved clinical literature study to ensure the implementation matches established protocols. This combination of technical innovation and clinical grounding is the project's unique contribution.

April 20 – 26, 2026 Week 8

Clinical Reasoning Engine & Emergency Condition Handling

Development Milestone Bug Fix

Work Done

The clinical reasoning engine was implemented in RulesEngine.swift — the core diagnostic logic that analyses questionnaire responses to suggest wound etiology. The engine differentiates between venous ulcers, arterial ulcers, diabetic foot ulcers, pressure injuries, mixed etiology wounds, and rarer conditions using evidence-based algorithms.

The diagnostic logic follows clinical principles: checking rare but dangerous conditions (malignancy, calciphylaxis) before common diagnoses, mirroring real clinical reasoning. Location-based pathways use the anatomical wound location to narrow the differential diagnosis before other factors are considered.

Emergency banners were improved for prominent alerts on dangerous findings. The report now uses collapsible sections for progressive information disclosure. Character encoding issues with medical symbols (cm², bullet points, Slovak characters) were resolved. Auto-advance on single-select questions was removed to give users control over the assessment pace.

Problems Encountered

Character encoding corruption with special medical characters across localised strings. A UI glitch with white squares appearing intermittently. Auto-advance was confusing users who wanted to review selections before proceeding.

Solutions Found

Systematic UTF-8 encoding verification across all string files. Traced the white square glitch to empty image placeholders and added fallback views. Removed auto-advance entirely — explicit "Next" taps give users control.

Personal Development

Building the reasoning engine required extensive study of wound etiology determination. Learned the hierarchical approach: rule out dangerous conditions first, then consider location probabilities, then evaluate specific signs. This "safety-first" logic is fundamental to clinical decision support.

April 27 – May 3, 2026 Week 9

Security Implementation & Rules Engine Refinement

Development Milestone Research

Work Done

Implemented the comprehensive security layer required for a healthcare application. This includes Face ID/Touch ID via iOS LocalAuthentication, secure credential storage in the iOS Keychain with hardware encryption, a 15-minute auto-logout via InactivityManager.swift, and rate limiting. These were built as dedicated services: BiometricAuthManager.swift, KeychainManager.swift, RateLimiter.swift, and SecureLogger.swift.

The rules engine was refined with rare condition handling. A progressive disclosure toggle hides uncommon wound types (radiation wounds, calciphylaxis, pyoderma gangrenosum) by default while keeping them accessible. The radiation site assessment was fixed, medical disclaimers were added throughout, and Firestore storage paths were finalised. A bug preventing patient and wound group deletion was resolved.

Continued work on the project report — documented the requirements gathering process, the design rationale for the security architecture, and began writing the implementation section covering the rules engine approach.

Problems Encountered

Face ID required handling multiple edge cases: unsupported device, unenrolled biometrics, denied permissions, authentication failure. Firestore security rules didn't match actual document paths, causing permission errors on deletion. Balancing rare condition visibility with usability was tricky.

Solutions Found

Built BiometricAuthManager with comprehensive LAError handling and graceful password fallback. Audited every read/write operation against the security rules to align paths. Progressive disclosure keeps rare conditions hidden by default but accessible via toggle.

Personal Development

Healthcare security features aren't optional — auto-logout, biometric auth, and encrypted storage are essential for patient data. Studied GDPR Article 9 requirements for health data processing, which directly informed the security architecture. Reviewed the implementation with my supervisor.

May 4 – 10, 2026 Week 10

Compression Safety, GDPR Compliance & Final Development

Development Milestone Research

Work Done

Implemented the critical compression therapy safety logic — arguably the most important safety feature in the entire application. The rules engine now ensures compression is never recommended without confirmed adequate arterial flow (ABI > 0.8). Compression applied to limbs with poor circulation can cause tissue necrosis, making this a genuine patient safety requirement.

The GDPR-compliant account deletion feature (UserDeletion.swift) was built with full cascade deletion: patient records, wound groups, assessments, and photographs are permanently removed from both Firestore and Firebase Storage. A compliance notice (ComplianceNoticeSheet.swift) was added for first-time users. The auto-logout was hardened for proper background/foreground transitions.

Peripheral neuropathy fields were removed from the questionnaire after determining they added friction without improving diagnostic output. The dressing engine, safety card, and report sections received final refinements. Remaining localised strings were added to ensure complete bilingual coverage.

With core development wrapping up, dedicated more time to the project report this week — documenting the compression safety logic, GDPR implementation decisions, and the clinical reasoning architecture. Also created system architecture diagrams using Lucidchart for inclusion in the report.

Problems Encountered

Compression logic needed to handle ambiguous cases — unmeasured or borderline ABI. Cascade deletion in Firestore requires manual traversal since there's no built-in cascade delete for subcollections. Auto-logout timer had edge cases where it wouldn't reset after certain interactions.

Solutions Found

Conservative compression approach: if ABI is unknown or borderline, the system recommends vascular referral rather than suggesting compression — "when in doubt, don't compress." Built a recursive deletion service traversing the full document hierarchy. Fixed auto-logout by hooking into UIApplication touch events to detect all user interactions.

Personal Development

The compression safety work was informed by extensive consultation with my clinical advisor — understanding that incorrect compression can lead to limb loss made this feature feel critically important. Implementing GDPR's "right to erasure" in practice was more complex than theoretical understanding suggested. This week marked the transition from active development to the testing and documentation phase.
